System Architecture Document

Production-ready architecture for CareerTwin AI

The architecture prioritizes fast simulation loops, explainability, secure data handling, and long-term scalability from hackathon MVP to multi-tenant platform.

Technology Stack

Backend API Node.js + TypeScript (Fastify or NestJS) High throughput, strong typing, team velocity
AI Orchestration Python services (FastAPI) + task workers Model ecosystem compatibility and async compute
Primary Database PostgreSQL Relational integrity for users, versions, decisions

Frontend Architecture

  • App shell with authenticated route guards and role-aware layout rendering.
  • Domain modules: Onboarding, Twin Profile, Simulation, Insights, Reports, Settings.
  • State management split: server-state cache (query layer) + UI state store.
  • Design system tokens for typography, spacing, motion, and accessibility consistency.
  • Progressive hydration for heavy analytics and simulation modules.

Backend Architecture

Gateway Service

  • Request authentication and tenant routing.
  • Rate limiting and request normalization.
  • BFF endpoints for frontend optimized payloads.

Core Domain Services

  • User & Profile Service
  • Twin Lifecycle Service
  • Simulation Service
  • Insights Service
  • Reporting Service

AI Services

  • Interview Orchestrator
  • Reasoning + Explainability Generator
  • Calibration and Contradiction Detector

Asynchronous Workers

  • Import Parsers (resume, LinkedIn, GitHub, portfolio)
  • Report generation and export
  • Weekly insights and scheduled recomputation

Database Architecture

PostgreSQL stores transactional and relational data; event streams and analytics snapshots feed downstream insights pipelines.

Authentication & Authorization

  • OIDC/OAuth2 with short-lived access tokens and rotating refresh tokens.
  • Session invalidation on security events and device-level session controls.
  • Role model: user, admin, support (least privilege by default).
  • Signed service-to-service auth for internal APIs (mTLS or token exchange).
  • Per-feature authorization checks at gateway and domain service levels.

Storage Layer

  • Object storage buckets for resumes, generated reports, and optional artifacts.
  • Content-addressed keys to prevent duplicate storage and support dedupe.
  • KMS-managed encryption keys and lifecycle retention rules.
  • Presigned URL strategy for controlled client-side upload/download.
  • Data tiering for cold archives of historical reports.

Caching Strategy

Redis Caches

  • Simulation result cache keyed by twin version + scenario hash.
  • Short TTL profile summary cache for dashboard speed.
  • Rate-limiting counters and idempotency token store.

Invalidation Rules

  • Invalidate on twin version change.
  • Invalidate insight cache on new simulation completion.
  • Use write-through for critical consistency flows.

API Layer

API style: versioned REST for core platform, async jobs for heavy compute, optional GraphQL read gateway for internal clients.

Endpoint Purpose Sync/Async
POST /v1/twin/build Create or refresh twin version Async job with polling
POST /v1/simulations/run Run scenario simulation Sync for light runs, async fallback for heavy runs
POST /v1/simulations/compare Return tradeoff comparison Sync
GET /v1/insights Fetch insight feed Sync
POST /v1/reports/generate Create export artifact Async

Data Flow

  1. User action hits API gateway and auth middleware.
  2. Gateway routes to domain service (twin, simulation, insight).
  3. Service resolves from cache first; falls back to DB/object store as needed.
  4. Heavy tasks publish to queue and worker processes perform async computation.
  5. Results persisted, cache updated, events emitted for downstream insights/reporting.

Microservices Strategy (if needed)

Use modular monolith in MVP to maximize speed; evolve to service boundaries when load, team size, and domain complexity justify operational overhead.

When to stay modular monolith

  • Single core team and rapid iteration requirements.
  • Low independent scaling pressure between domains.
  • Simpler deployment and debugging lifecycle.

When to split services

  • Simulation compute outgrows API latency budget.
  • Insights/report generation requires separate scaling behavior.
  • Clear ownership boundaries by team/domain emerge.

Deployment & DevOps

  • Environment tiers: dev, staging, production with isolated data boundaries.
  • CI/CD pipeline: lint, test, security scan, build, integration tests, deploy.
  • Blue/green or canary deployments for low-risk rollouts.
  • Feature flags for controlled releases of high-impact AI features.
  • Backup and disaster recovery policy for DB and object storage.
  • Runbook-driven incident response and SLO-based alerting.

Scalability & Reliability

Horizontal Scaling

  • Stateless API pods with autoscaling by CPU + latency.
  • Dedicated simulation worker pools with queue-based backpressure.
  • Read replicas for high read-heavy insight/report queries.

Reliability Controls

  • Circuit breakers on external providers.
  • Idempotency keys for simulation/report generation endpoints.
  • Graceful degradation to cached insights on partial outages.

Observability

  • Distributed traces across frontend, gateway, and services.
  • Model-quality dashboards for output confidence drift.
  • Error budgets linked to product-critical flows.

Data Governance

  • PII classification and scoped access controls.
  • Retention rules for generated artifacts and logs.
  • Compliance-ready audit trails for recommendation events.

Trade-offs & Key Decisions

Decision Chosen Option Trade-off
Architecture style Modular monolith first Faster early velocity, but future extraction effort required
API strategy REST-first with async jobs Simpler client integration, less flexible than full GraphQL mesh
Simulation compute Async workers for heavy runs Better reliability, added orchestration complexity
Data storage split Postgres + object storage + cache Clear responsibilities, more infrastructure surfaces to operate
Feature rollout Flag-gated progressive release Lower risk, requires robust flag governance
