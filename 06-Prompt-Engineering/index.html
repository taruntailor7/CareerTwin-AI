Prompt Engineering Specification

Reliable prompt architecture for CareerTwin AI

Prompting in CareerTwin AI is treated as production infrastructure. Prompts are modular, versioned, testable, and constrained by strict output schemas and safety policies.

System Prompts

System prompts define hard behavior boundaries and must never be overridden by user content.

[System Prompt: CareerTwin Core]
You are CareerTwin AI, a decision intelligence system.
Non-negotiable rules:
1) Never make deterministic future claims.
2) Always include assumptions, tradeoffs, confidence range.
3) Cite evidence source types for important claims.
4) If data is insufficient, ask clarifying questions before advising.
5) Output must follow JSON schema exactly.
6) Reject unsafe or manipulative requests and provide safe alternatives.

Developer Prompts

  • Task-scoped instructions per module: interview, simulation, insight generation, reporting.
  • Include domain context and output format contract in each prompt package.
  • Use explicit fallback behavior for low-confidence or missing-data cases.
[Developer Prompt: SimulationOrchestrator]
Goal: Compare 2-5 user scenarios and produce ranked tradeoff analysis.
Constraints:
- Use only provided context bundle.
- Do not invent salary values without source.
- Mark missing assumptions explicitly.
Output: SimulationComparisonSchema v1.3

Career Interview Prompts

Intent

Fill profile gaps and surface latent career constraints and preferences.

Prompt Pattern

One high-leverage question at a time with confidence-aware follow-ups.

[Interview Prompt Template]
Context: {profile_summary}, {known_constraints}, {confidence_gaps}
Task:
1) Ask exactly one clarifying question.
2) Keep under 25 words.
3) If prior answer is contradictory, ask reconciliation question.
Output JSON:
{
  "question": "...",
  "question_type": "constraint|motivation|preference|risk",
  "reason": "why this question now"
}

Simulation Prompts

  • Scenario prompts operate on normalized context bundles with explicit assumptions.
  • Every output must return quantitative ranges plus qualitative risk narrative.
  • Simulation prompts include stress-test mode for assumption sensitivity.
[Simulation Prompt Skeleton]
Inputs:
- twin_version
- scenario_set[]
- market_context
- user_constraints

Tasks:
1) Evaluate each scenario across required metrics.
2) Compute confidence band per metric.
3) Produce tradeoff summary and next-step experiments.

Reasoning Prompts

  • Reasoning stage converts raw outputs into user-readable decision intelligence.
  • Requires explicit separation of facts, assumptions, and inferred conclusions.
  • Must include uncertainty language and actionability.
[Reasoning Prompt Contract]
Produce:
1) Recommendation headline
2) Assumptions
3) Evidence-linked rationale
4) Tradeoffs
5) Confidence range
6) 3 action steps
Disallow:
- certainty language ("will definitely", "guaranteed")
- unsupported numerical claims

JSON Schemas

All model outputs are schema-validated before downstream use.

{
  "$id": "SimulationComparisonSchema-v1.3",
  "type": "object",
  "required": ["scenarios", "summary", "recommendation", "confidence_band", "evidence_refs"],
  "properties": {
    "scenarios": {
      "type": "array",
      "minItems": 2,
      "items": {
        "type": "object",
        "required": ["name", "metrics", "tradeoffs"],
        "properties": {
          "name": {"type": "string"},
          "metrics": {"type": "object"},
          "tradeoffs": {"type": "array", "items": {"type": "string"}}
        }
      }
    },
    "summary": {"type": "string"},
    "recommendation": {"type": "string"},
    "confidence_band": {
      "type": "object",
      "required": ["min", "max"],
      "properties": {"min": {"type": "number"}, "max": {"type": "number"}}
    },
    "evidence_refs": {"type": "array", "items": {"type": "object"}}
  }
}

Structured Outputs

Output Type Schema Validation Action
Interview Question InterviewQuestionSchema Reject and retry on missing reason/question_type

Insight Card InsightCardSchema Drop non-critical fields, keep required core

Prompt Templates

  • Template variables: user profile, twin version, scenario list, market context, safety mode.
  • Guardrail blocks are always prepended.
  • Template engine supports conditional sections for low/high confidence modes.
Template: InsightsWeeklyDigest-v2
Sections:
- Role
- Goals
- Constraint shifts
- Top opportunities
- Top risks
- Action suggestions
- Confidence explanation

Prompt Versioning

  • Semantic versioning per prompt family (major/minor/patch).
  • Prompt registry stores prompt text, schema ref, owner, rollout status, eval score.
  • Canary rollout with automatic rollback on quality regression.

Prompt Evaluation

Offline

  • Golden prompts with expected structured outputs.
  • Rubric scoring: relevance, accuracy, explainability, safety.
  • Regression suite run on every prompt change.

Online

  • A/B prompt experiments by user cohort.
  • Measure confidence calibration and correction rate.
  • Capture user override and dissatisfaction signals.

Failure Cases

  • Schema violation (missing required fields).
  • Unsupported claims without evidence references.
  • Conflicting recommendations across similar contexts.
  • Overconfident language in low-data scenarios.
  • Prompt injection from imported untrusted content.

Recovery Strategies

  1. Automatic retry with constrained prompt variant.
  2. Fallback prompt with reduced scope and stricter schema.
  3. Safe-mode summary with explicit uncertainty.
  4. Escalation to deterministic rules engine for critical outputs.
  5. Event logging for eval loop and prompt refinement backlog.

Examples

Good Output

{
  "recommendation": "Prioritize AI platform pivot with staged upskilling.",
  "assumptions": ["6 hrs/week learning", "no relocation in 12 months"],
  "confidence_band": {"min": 0.58, "max": 0.73},
  "tradeoffs": ["Short-term salary volatility", "Higher long-term optionality"]
}

Bad Output

"You should definitely switch jobs now. It will increase salary by 40%."
# Issues:
# - Deterministic claim
# - No assumptions
# - No evidence
# - No confidence band

Best Practices

  • Keep prompt roles explicit and single-purpose.
  • Always bind outputs to strict schemas.
  • Prefer short, deterministic instructions over verbose ambiguity.
  • Treat prompt text as versioned code with tests.
  • Measure prompt quality continuously, not only at release.
  • Separate untrusted content from instruction channels.
